## 2. Web display was the real gap

The user opened http://localhost:9292/ during the run and saw nothing live.
rack-test (Task 5) covered the routes in isolation but never the wired
pipeline: `fluentd → POST /_internal/notify → WEBSOCKETS broadcast →
browser WebSocket → DOM render`. chrome-mcp was used to inspect the
running page directly.

### Defect 1 — three.js double-load crashed `ForceGraph3D` init

`web/views/index.erb` loaded `three@0.160.0` standalone and then
`3d-force-graph` (which UMD-bundles its own three.js).
Console captured:

```
[WARNING] THREE.WARNING: Multiple instances of Three.js being imported.
[EXCEPTION] Uncaught TypeError: Ak.Timer is not a constructor
[ERROR] NoMethodError: undefined method 'new' for NilClass  (picoruby)
```

Because `JS.global[:ForceGraph3D]` stayed undefined, the very first line
of `app.rb` that touched it (`...new.call(getElementById('graph-canvas'))`)
threw, and **every subsequent line** (WebSocket open, /api/recent fetch)
never ran. The static HTML rendered Quick / Perfect / Graph headings only.

**Fix:** removed the standalone three.js `<script>` tag — the bundled copy
inside 3d-force-graph is sufficient.

### Defect 2 — even with three deduped, 3d-force-graph CDN init still throws

After removing the duplicate three load:

```
[ERROR] TypeError: Cannot set properties of undefined (setting 'innerHTML')
        at g.init (3d-force-graph)
[ERROR] NoMethodError: undefined method 'nodeAutoColorBy' for NilClass
```

This is an issue inside the current 3d-force-graph CDN bundle's
container-init logic, independent of how it's invoked from PicoRuby.
The graph panel is *visualization on top of* the core "Web 表示" feature,
not the feature itself; transcripts must render whether or not the graph
works.

**Fix (deferred resolution):** disabled the graph init — `GRAPH = nil` —
and gated `redraw_graph` on it. `app.rb` proceeds past the graph block,
opens the WebSocket, fetches `/api/recent`. **TODO** pinned in the source:
find a working three.js + 3d-force-graph version pair and re-enable.

### Defect 3 — PicoRuby JS bridge mismatches

Two separate API mismatches blocked all data flow even after defect 2 was
gated:

- `JS.global.fetch(url).then { |resp| ... }` raised
  `ArgumentError: JS::Object#fetch requires a block: use 'fetch(url) { |resp| ... }'`.
  PicoRuby's JS bridge shims `fetch` to take a block and resolve the
  Promise internally. **Fix:** use the block form. Note `Response#json()`
  is *not* shimmed — it still returns a JS Promise, so the inner call
  needs `resp.json.then do |data| ... end`.

- The WebSocket message handler used
  `JSON.parse(...).to_a` followed by `msg.find { |kv| kv[0].to_s == 'type' }`.
  Even though messages were arriving (chrome-mcp logged 194 events
  including `{"type":"quick","data":{...}}` from real screen audio),
  `find` consistently returned nil and the handler skipped via
  `next unless type`. **Fix:** direct property access — `parsed[:type]` /
  `parsed[:data]` — works correctly through the bridge.

## 4. Test-suite flake (unrelated but caught here)

`test_stop_via_pidfile_aborts_when_term_ignored` flaked when run inside
the full `rake test` suite (passed when run alone). Root cause: the
fixture reuses fixed pidfile names (`lifecycle-test.pid`, `stubborn.pid`)
and a leftover from a prior run lets the
`20.times { break if File.exist?(pidfile); ... }` wait return on
iteration 0 — before the freshly-spawned child has written its own pid.
`stop_via_pidfile` then reads the stale-and-dead pid and takes the
"stale pidfile" branch, returning normally instead of aborting.

**Fix:** test setup/teardown explicitly delete both pidfiles so each test
gets a clean slate. Verified stable across 3 consecutive `rake test`
runs (23 tests, 51 assertions, 0 failures each time).
